# Git Rules — Three Card Survival

## 브랜치 구조

```
master  ← 안정적으로 플레이 가능한 버전만
dev     ← 평소 작업 브랜치 (기본 작업 위치)
feature/<name>  ← 큰 기능 단위 (선택적)
fix/<name>      ← 긴급 버그픽스 (선택적)
```

## 브랜치 규칙

### master
- 직접 커밋 금지
- `dev`에서 기능이 완성되고 플레이 가능한 상태일 때만 merge
- merge 시 반드시 플레이테스트 통과 확인 후 진행

### dev
- 평소 모든 작업은 여기서 진행
- 작업 세션 끝마다 push
- 깨진 상태(컴파일 에러 등)로 push하지 않음

### feature / fix 브랜치 (선택적)
- 2일 이상 걸리거나 기존 기능을 크게 건드리는 작업에만 사용
- `git checkout -b feature/<name> dev` 로 생성
- 완료 후 `dev`에 merge, 브랜치 삭제

## 커밋 메시지 형식

```
<변경 요약> (한 줄, 한국어 또는 영어)

- 세부 변경 1
- 세부 변경 2
```

- 첫 줄 50자 이내
- 여러 변경사항은 `-` 리스트로 나열
- 스크립트 변경 / 데이터 에셋 변경 / 버그 픽스는 구분해서 기술

## dev → master merge 조건

다음을 모두 만족할 때:
- [ ] 컴파일 에러 없음
- [ ] 핵심 게임 루프 (카드 선택 → 하루 종료 → 보상) 정상 동작
- [ ] 게임오버 / 클리어 조건 정상 동작

## Claude가 push할 때 따르는 절차

1. 현재 브랜치 확인 → `dev`인지 체크
2. `dev`가 아니면 `dev`로 이동 후 작업
3. `.claude/settings.local.json` 제외하고 스테이징
4. 커밋 메시지 위 형식대로 작성
5. `git push origin dev`
6. master merge는 사용자가 명시적으로 요청할 때만 진행
